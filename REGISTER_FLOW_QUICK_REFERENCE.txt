Q: Are the database INSERTs the same?
A: NO. Register = 1 INSERT (users), EndUser = 2 INSERTs (users + user_orgs).
   Phone field differs critically: Register populated, EndUser empty string.
   See: ANALYSIS.A

Q: What's the biggest conflict?
A: THREE major conflicts:
   1. Phone: Register requires it, EndUser discards it
   2. Username: Register 3-32 strict, EndUser 3-64 permissive
   3. Password: Register string hash, EndUser VO pattern
   See: SUMMARY "Critical Merge Decisions"

Q: What validations must I understand?
A: Five key validators:
   1. ValidateUserName (3-32, start [a-zA-Z_-], reserved) - user domain
   2. ValidateUsername (3-64, any start, no reserved) - enduser domain
   3. NewPhoneNumber (11-digit China format) - user domain only
   4. ValidatePasswordStrength (8+ chars, letter, digit) - both domains
   5. Merged validator: adopt stricter (user's) rules
   See: ANALYSIS.B, C, D, E

Q: What fields am I adding?
A: FOUR fields being added to merged User:
   1. OrgName (string) - tenant scope key
   2. IsForbidden (bool) - account disable flag
   3. IsAdmin (bool) - org admin flag
   4. Password (HashedPassword VO) - replaces PasswordHash string
   See: CHECKLIST.G

Q: What can I delete?
A: FOUR items can be removed from domain/user:
   1. registerNameAdjectives (marked unused)
   2. registerNameNouns (marked unused)
   3. generateRegisterDisplayName() (unused)
   4. NewOAuthUser() (consider deprecating)
   See: SUMMARY "Cleanup Opportunities"

Q: What methods must I add?
A: FOUR methods from EndUser → merged User:
   1. Enable() - set IsForbidden=false
   2. Disable() - set IsForbidden=true
   3. IsActive() - return !IsForbidden
   4. VerifyPassword() - delegate to Password.Verify()
   See: CHECKLIST.H

Q: How many phases of work?
A: FIVE phases:
   Phase 1: Schema (add 3 columns)
   Phase 2: Domain (merge struct, add methods)
   Phase 3: Validation (unify rules)
   Phase 4: Repository (update Create)
   Phase 5: Cleanup (remove unused code)
   See: SUMMARY "Implementation Priority"

Q: What's the new User struct look like?
A: 10 fields (was 7, EndUser had 8):
   ID, ExternalID, OrgName, Name, Phone, Password, IsForbidden, IsAdmin,
   CreatedAt, UpdatedAt
   See: CHECKLIST.G for full struct

⚠️  CRITICAL RISKS
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

RISK 1: Stricter username validation breaks existing EndUsers
  SEVERITY: MEDIUM
  MITIGATION: Pre-migration audit of all existing usernames
  See: CHECKLIST "Risk Assessment"

RISK 2: Phone field handling ambiguous for tenant-only users
  SEVERITY: HIGH
  MITIGATION: Clear rules: required for phone-based register, empty for tenant
  See: CHECKLIST "Risk Assessment"

RISK 3: Password VO migration requires rewriting all login code
  SEVERITY: MEDIUM
  MITIGATION: HashedPassword.NewHashedPasswordFromHash() = no re-hashing needed
  See: ANALYSIS.E

RISK 4: user_orgs table becomes redundant, data sync issues
  SEVERITY: LOW
  MITIGATION: Keep user_orgs temporarily, gradual deprecation
  See: CHECKLIST "Risk Assessment"
