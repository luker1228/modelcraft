# 受约束 GraphQL 运行时专利方案设计

**日期**: 2026-06-07  
**状态**: Approved  
**范围**: 基于 ModelCraft 现有设计态/运行态架构，提炼一项可申请的发明专利方案

---

## 1. Summary

本方案不保护 GraphQL 本身，也不保护“低代码平台”这一宽泛产品概念。

本方案拟保护的是一种**基于模型元数据和权限策略动态生成受约束 GraphQL 访问边界，并通过语义反馈形成查询修正闭环的数据访问控制方法**。

该方案的核心目标不是发明新的查询语言，而是在保留现有 GraphQL 作为查询承载协议的前提下，解决普通数据平台常见的三个问题：

1. 开放自由 SQL 或近似自由查询带来的注入与越权风险
2. 权限系统只能停留在表级、接口级或粗粒度字段级，难以细化到查询条件和操作符层
3. AI 在面对动态数据接口时不知道“什么能查、怎么查、查错了如何改”

本方案的推荐申请路线为：

- 主申请聚焦“运行时受约束控制机制”
- AI native 作为该机制带来的技术效果与实施优势

---

## 2. Problem

普通数据平台通常存在以下结构性问题：

### 2.1 自由查询不安全

若平台允许用户直接编写 SQL、拼接条件或通过接近自由表达的查询接口访问数据，则难以同时控制：

- SQL 注入风险
- 越权字段访问
- 越权条件构造
- 非授权聚合、排序、关联展开

### 2.2 权限系统不够细

很多系统即使支持权限控制，也只停留在：

- 是否能访问某张表
- 是否能看到某个字段
- 是否能调用某个接口

但无法进一步约束：

- 某字段是否允许作为过滤条件
- 某字段允许哪些查询操作符
- 某字段是否允许排序
- 是否允许聚合、统计、分组
- 是否允许多层级关联访问

### 2.3 AI 无法稳定构造合法查询

在 AI 或 Agent 场景中，若系统没有显式暴露能力边界，AI 只能猜测可用字段、过滤方式和返回结构。

即使错误信息对人类可读，普通用户仍需要掌握 GraphQL 知识才能修正；而如果错误语义足够明确，AI 即使不依赖专用协议，也能据此自动修正查询并逐步收敛到合法请求。

本方案的关键判断是：

> 不是“先有 AI，再做能力描述”，而是“先生成受约束访问边界，AI 才能基于该边界稳定生成合法查询”。

## 3. Goals

本专利方案需要同时实现以下目标：

1. 基于模型元数据生成运行时 GraphQL 访问能力
2. 基于权限策略对运行时能力进行主体专属裁剪
3. 将访问控制细化到字段、条件、操作符、结果结构和关联访问层
4. 当查询超出能力边界时返回语义明确的反馈信息
5. 使调用方能够根据反馈修正查询，形成访问闭环
6. 在不开放自由 SQL 的情况下完成数据访问
7. 让 AI 能利用该闭环稳定构造合法 GraphQL 请求

---

## 4. Non-Goals

本方案不尝试保护以下内容：

- GraphQL 标准语法本身
- GraphQL 的内建类型校验能力
- 任意“后台可把请求转换为安全执行计划”这一泛化表述
- 专门为 AI 设计的新错误码协议
- 通用聊天助手、提示工程或自然语言生成本身
- 低代码建模平台的全部产品能力

原因如下：

1. GraphQL 是开源和现有协议，不能作为本发明核心
2. 类型约束属于 GraphQL 原生能力，不能视为创新点
3. 仅写“安全执行”过于常见，难以构成尖锐差异点
4. 本方案的创新不是 AI 协议，而是**边界生成 + 语义反馈 + 查询修正**这一闭环

## 6. Core Insight

本方案的发明点不在于发明一种新的查询语言，而在于：

1. 使用模型元数据描述业务对象和查询能力属性
2. 使用权限策略描述主体可访问的查询范围
3. 将两者联合编译为**主体专属受约束 GraphQL 访问边界**
4. 当调用方提交超界查询时，返回指向超界位置和超界原因的语义反馈
5. 使调用方能够据此修正查询，直至形成合法请求

因此，该方案解决的是一个“受约束运行时数据访问闭环”问题，而不是单纯的接口暴露问题。

## 7. System Design

该方案可拆分为 5 个协同模块。

### 7.1 模型定义模块

负责维护业务对象、字段、字段类型、对象关系及各字段对应的查询能力属性。

这里的查询能力属性可包括：

- 是否允许返回
- 是否允许作为过滤条件
- 允许哪些过滤操作符
- 是否允许排序
- 是否允许聚合
- 是否允许关联展开

### 7.2 权限策略模块

负责维护调用主体对应的访问策略。

该访问策略至少约束到：

- 字段级
- 操作级
- 条件级
- 结果结构级

其中“条件级”不是简单字段可见，而是进一步约束：

- 某字段是否能参与过滤
- 能参与哪些操作符
- 是否允许排序或聚合

### 7.3 接口生成模块

负责根据模型元数据生成运行态 GraphQL 能力，并结合权限策略对 GraphQL 能力进行裁剪，形成主体专属的受约束接口视图。

创新重点不在 GraphQL 语法，而在于：

- 动态生成
- 按策略裁剪
- 输出可访问能力边界

### 7.4 运行时访问控制模块

负责在运行时只接受落在受约束边界内的 GraphQL 请求。

这里不是主创新点，但作为闭环组成部分仍然必要。其职责是确保调用方不能绕过受约束访问边界直接形成自由查询。

### 7.5 语义反馈模块

当请求超出能力边界时，系统不只返回泛化失败，而是返回语义明确的反馈信息。

该反馈既可供熟悉 GraphQL 的人类手工修正，也可供 AI 自动理解并重构下一次查询。

本方案不要求反馈必须是专门 AI 协议；只要反馈语义足够清晰，即可支持 AI 自动修正。

---

## 8. Closed Loop

本方案区别于普通 GraphQL 平台或普通权限平台的关键，在于形成如下闭环：

1. 获取模型元数据与权限策略
2. 生成主体专属受约束 GraphQL 访问边界
3. 向调用方提供该访问边界，用于指导查询构造
4. 当调用方提交超界查询时，返回语义反馈
5. 调用方依据反馈修正查询
6. 最终提交合法查询并完成数据访问

该闭环的保护重点不是“GraphQL 会做类型校验”，也不是“后台会执行安全查询”，而是：

- 访问边界是动态生成的
- 访问边界细化到字段、条件、操作符、结果结构和关联访问层
- 反馈语义可指向具体违规点
- 反馈能够驱动查询修正

## 9. Method Flow

建议主方法权利要求围绕以下步骤组织。

### 9.1 获取模型元数据

读取目标数据对象的模型定义，所述模型定义至少包括：

- 字段定义
- 字段类型
- 对象关系
- 字段对应的查询能力属性

### 9.2 获取权限策略

根据用户、角色、租户、项目或其他身份上下文，确定调用主体对目标数据对象的访问约束。

该访问约束至少覆盖：

- 字段访问范围
- 查询操作范围
- 条件使用范围
- 返回结果范围

### 9.3 生成受约束 GraphQL 访问边界

基于模型元数据生成目标数据对象对应的 GraphQL 访问能力，并基于权限策略对该访问能力进行裁剪，得到面向该调用主体的受约束 GraphQL 访问边界。

该边界至少体现：

- 允许访问哪些字段
- 哪些字段允许作为过滤条件
- 每个字段允许哪些查询操作符
- 是否允许排序、分页、聚合、关联展开

### 9.4 暴露访问边界

向调用方提供所述受约束 GraphQL 访问边界，用于指导调用方构造 GraphQL 查询请求。

这里的调用方既可以是普通客户端，也可以是 AI 或 Agent。

### 9.5 返回超界语义反馈

当调用方提交的 GraphQL 查询请求超出所述受约束 GraphQL 访问边界时，输出指示超界位置及超界原因的语义反馈信息。

所述反馈不要求是专门机器协议，但应当具有明确语义，使其既可供熟悉 GraphQL 的人类理解，也可供 AI 自动利用。

### 9.6 完成合法查询访问

在接收到满足所述受约束 GraphQL 访问边界的查询请求时，执行对应的数据访问并返回结果。

该步骤是闭环收尾，不作为主创新点单独强调。

## 10. Independent Claim Strategy

推荐采用“边界生成 + 反馈闭环联合型”作为独立权利要求 1 的骨架。

### 10.1 推荐骨架

一种基于模型元数据和权限策略生成受约束 GraphQL 访问能力并形成查询修正闭环的数据访问控制方法，其特征在于，包括：

1. 获取目标数据对象的模型元数据，所述模型元数据至少包括字段定义、字段类型、对象关系以及字段对应的查询能力属性；
2. 获取与调用主体对应的权限策略，所述权限策略用于限定所述调用主体对目标数据对象的字段访问范围、查询操作范围和条件使用范围；
3. 基于所述模型元数据生成目标数据对象对应的 GraphQL 访问能力，并基于所述权限策略对所述 GraphQL 访问能力进行裁剪，得到面向所述调用主体的受约束 GraphQL 访问边界；
4. 向调用方提供所述受约束 GraphQL 访问边界，用于指导调用方构造针对目标数据对象的 GraphQL 查询请求；
5. 在接收到所述 GraphQL 查询请求超出所述受约束 GraphQL 访问边界时，输出指示超界位置及超界原因的语义反馈信息，所述语义反馈信息用于指导调用方修正所述 GraphQL 查询请求；
6. 在接收到满足所述受约束 GraphQL 访问边界的 GraphQL 查询请求时，执行对应的数据访问操作并返回查询结果。

### 10.2 核心保护点

上述独立权利要求中，真正的发明中心在于：

- 主体专属访问边界的动态生成
- 越界查询的语义反馈
- 二者组合形成查询修正闭环

其中，GraphQL 只是承载协议，而不是创新主体。

## 11. Dependent Claim Coverage

建议至少通过从属权利要求覆盖以下细节。

### 11.1 字段级约束

受约束 GraphQL 访问边界限定调用主体可访问的字段集合，不同调用主体对应不同字段访问范围。

### 11.2 条件级约束

并非所有可见字段都可作为过滤条件；系统进一步限定哪些字段可用于查询条件构造。

### 11.3 操作符级约束

针对可过滤字段，进一步限定允许的操作符集合。

### 11.4 关联访问约束

针对关系字段，限定是否允许关联展开、允许展开的目标对象及允许展开的层级深度。

### 11.5 结果结构约束

限定是否允许排序、分页、聚合、分组或统计类返回结构。

### 11.6 主体专属能力生成

受约束 GraphQL 访问边界不是固定 schema，而是根据主体身份、租户、项目、角色或资源上下文动态裁剪生成。

### 11.7 反馈内容

当查询超界时，反馈信息至少包含以下之一：

- 违规字段
- 违规条件
- 违规操作符
- 违规关联路径
- 缺失权限类型
- 可替代查询方式

### 11.8 AI 适配性

语义反馈信息采用对人类可理解且语义明确的表达形式，使熟悉 GraphQL 的用户可手工修正查询，并使 AI 能基于所述反馈重构合法请求。

### 11.9 模型驱动生成

字段可过滤性、可排序性、可聚合性来源于模型元数据中的查询能力属性，而不是人工硬编码到接口实现中。

### 11.10 权限与模型联合裁剪

访问边界由模型元数据与权限策略共同决定，而不是单独靠 schema 定义或单独靠后置鉴权。

## 12. Key Distinctions From Existing Platforms

建议在专利背景技术与发明内容中反复强调以下区别。

### 12.1 与自由 SQL 平台的区别

现有平台通常允许自由 SQL、SQL 方言或近似自由表达，难以同时约束：

- 注入风险
- 条件级权限边界
- 非授权结果结构

本方案以受约束 GraphQL 访问边界替代自由 SQL 暴露方式。

### 12.2 与粗粒度权限平台的区别

现有平台即使支持权限，也多停留在表级或字段可见级，难以约束“某字段能否作为过滤条件、支持哪些操作符、是否允许排序/聚合/关联展开”。

本方案将权限控制推进到查询条件和操作符层。

### 12.3 与普通 GraphQL 平台的区别

现有 GraphQL 平台通常只解决接口暴露问题，不形成“主体专属能力边界 + 超界反馈 + 查询修正”的访问闭环。

### 12.4 与 AI 助手方案的区别

现有 AI 助手大多通过提示或经验猜测接口能力；而本方案先生成并暴露访问边界，再通过语义反馈支持查询收敛，因此 AI 可用性来源于受约束运行时机制，而非单独的提示工程。
