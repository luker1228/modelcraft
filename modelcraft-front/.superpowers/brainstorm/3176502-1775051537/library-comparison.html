基于 ModelCraft 的实际需求分析

react-hook-form

本质定位

表单状态管理 + 校验库。不管 UI 渲染，你还是要自己写每个字段的组件。

接入方式
// 你还是要手写字段渲染 {fields.map(field => ( // 自己判断类型 → 自己写组件 field.type === 'date' ? <DatePicker {...register(field.name)} /> : <Input {...register(field.name)} /> ))}
与项目适配度
中等适配

JSON Schema → UI 的映射还是要自己写。但校验、错误处理会更规范。只解决了一半问题。

Formily
重量级 · 阿里出品
评分
上手难度
难
动态表单支持
极强
Schema 驱动
原生支持
生态/社区
★★★☆☆
本质定位

完整的低代码表单引擎。有自己的 schema 格式（不是标准 JSON Schema），需要转换。

接入方式
// 需要先把 JSON Schema 转为 Formily Schema // schema-transformer.ts 里有这个逻辑 const formilySchema = transformToFormily(jsonSchema) <SchemaField schema={formilySchema} /> // ↑ 自动渲染所有字段
与项目适配度
中等适配

schema-transformer.ts 里有 Formily 相关代码，但引入的是一套完整生态，概念多、bundle 大，当前需求用不到 80% 的功能。

react-jsonschema-form
⭐ 推荐 · 中量级
评分
上手难度
中等
动态表单支持
强
Schema 驱动
原生 JSON Schema
生态/社区
★★★★☆
本质定位

JSON Schema → 表单的直接映射。服务端返回什么 schema，前端直接渲染，零转换。

接入方式
// 服务端 JSON Schema 直接用！ const { data } = useQuery(MODEL_JSON_SCHEMA_QUERY) <Form schema={data.jsonSchema} ← 直接传入 uiSchema={uiSchema} ← 控制 UI 细节 onSubmit={handleSubmit} widgets={{ relation: RelationPicker }} />
与项目适配度
高度适配

项目已有 MODEL_JSON_SCHEMA_QUERY，返回标准 JSON Schema，RJSF 直接消费，无需任何转换层。自定义 widget 可插入 RelationPicker。

💡 综合结论

react-hook-form：只解决校验，UI 渲染还是要自己写，问题解决了一半。
Formily：功能最强，但有自己的 schema 格式，需要写转换层，且项目目前用不到它的高级特性。
RJSF：和项目已有的 JSON Schema 天然契合，直接对接服务端数据，支持自定义 widget（RelationPicker 就可以作为自定义 widget 插入），是三者中改动最少、收益最大的选择。
