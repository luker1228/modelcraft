动态表单 — 3 种方案对比

选择你倾向的实现方向（可点击卡片查看详情）

A

轻量改造（推荐）

在现有 DynamicModelTable 内部，将手写 if-else 表单替换为 DynamicFormField 组件，同时在字段 mutation 加上 refetchQueries。

优点

  • 改动范围小，风险低
  • 复用现有 schema-transformer.ts
  • 不引入新依赖
  • 1-2天可完成

缺点

  • DynamicModelTable 文件已经很大，继续增大
  • RELATION 选择器需要自己写
B

独立表单模块

将表单逻辑完全抽离到独立的 ModelRecordForm 模块，包含 DynamicFormField、RelationPicker、useFormSchema hook，DynamicModelTable 只负责表格展示。

优点

  • 职责清晰，表格与表单解耦
  • 表单可在其他地方复用
  • 文件结构更清晰

缺点

  • 需要明确接口设计
  • 3-4天完成
C

引入 Formily / react-hook-form

引入成熟的 schema-driven 表单库（如 Formily），JSON Schema 直接驱动表单渲染，项目中已有 schema-transformer 可对接。

优点

  • 功能强大（联动、条件显示）
  • schema-transformer.ts 已有 Formily 相关逻辑

缺点

  • 引入重型依赖
  • 学习成本高
  • 当前需求不需要联动等复杂特性
  • 5-7天完成
