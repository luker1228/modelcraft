# Quick Task 260505-2i9: 管理端 / 用户端双 UI 架构设计

## 背景

用户当前的设计困境根源：**试图用一套 UI 同时服务两类完全不同的受众**。

- **管理端（平台管理员 / Developer）**：配置模型、管理 EndUser、管理 RBAC，登录入口 `/login`
- **用户端（EndUser）**：使用 Workspace 操作数据，登录入口 `/end-user/{orgSlug}/login`

两者登录入口不同、权限体系不同、功能集合不同，强行共用 UI 导致设计割裂、逻辑耦合。

**结论：按两套 UI 设计，彻底解除耦合。**

**新增章节内容草稿:**

```markdown
## UI 架构决策

**核心原则：管理端和用户端是两套独立 UI，不共享主导航结构。**

### 管理端（Admin/Developer UI）

- **入口:** `/login` → 平台统一登录
- **受众:** 平台管理员、开发者
- **功能:** 模型设计、字段管理、RBAC 配置、EndUser 账号管理、数据库集群管理
- **路由前缀:** `/org/{orgName}/...`（现有结构保持）
- **特征:** 功能密集、配置导向、设计时操作

### 用户端（EndUser/Workspace UI）

- **入口:** `/end-user/{orgName}/login`（或更简洁的 `/{orgName}/login`）
- **受众:** 终端用户（End User）
- **功能:** 登录 → 选择项目 → 进入 Workspace 操作数据（CRUD）
- **路由前缀:** `/workspace/{orgName}/{projectSlug}/`
- **特征:** 精简、数据操作导向、运行时视图

### 为什么不用一套 UI

| 维度 | 管理端 | 用户端 |
|------|--------|--------|
| 登录入口 | `/login` | `/end-user/{orgName}/login` |
| Token 体系 | `scope=org`（统一后）| `scope=org` → exchange → `scope=project` |
| 功能复杂度 | 高（设计时） | 低（运行时 CRUD） |
| 导航结构 | 多层级侧边栏 | 扁平 Workspace 面板 |
| 权限来源 | RBAC（开发者角色）| RBAC PermissionBundle（EndUser 角色）|

两者**功能集合不重叠**，强行合并只会让两端都难用。
```
