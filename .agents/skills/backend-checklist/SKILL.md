# backend-checklist

错题本 = 真实发生过的 Bug 的结构化记录，每条记录末尾都有一条可操作的 **Checklist 规则**。
这个 skill 做两件事：往里写，往外用。

---

## 错题本位置

```
ai-metadata/backend/common-mistakes.md
```

始终从这里读写，不要创建其他文件。

---

## 能力一：add —— 加入错题本

**触发词**：「加入错题本」、「记录这个错误」、「把这个 bug 记下来」

### 步骤

1. **读取** `ai-metadata/backend/common-mistakes.md`，找到当前最大编号，确定新编号。
   编号格式：`BM-YYYYMMDD-XXXX`（今天日期 + 4位顺序号）。

2. **从对话上下文提取** 以下信息（不足时向用户确认）：
   - **问题描述**：一句话说清是什么错误
   - **错误代码**：❌ 错误写法 vs ✅ 正确写法（代码块）
   - **根因**：为什么会出这个问题（技术层面）
   - **症状**：如何表现/如何发现（错误消息、日志特征、用户侧现象）
   - **修复范围**：需要改哪几处
   - **Checklist 规则**：一条可直接复用的、以「凡是…必须…」或「禁止…」开头的审查规则

3. **追加** 到 `common-mistakes.md` 末尾，使用标准模板（见下方）。

4. **确认** 写入成功，告诉用户编号。

### 标准模板

```markdown
---

### {编号} · {简短标题}

**严重程度**: CRITICAL | HIGH | MEDIUM | LOW
**分类**: {data-isolation | sql-query | null-safety | error-handling | api-design | ...}
**发现日期**: {YYYY-MM-DD}

#### 问题描述

{一段话描述问题}

#### 错误代码

\```{语言}
// ❌ 错误
{错误写法}
\```

\```{语言}
// ✅ 正确
{正确写法}
\```

#### 根因

{技术根因说明}

#### 症状

{如何发现 / 错误消息 / 日志特征}

#### 修复范围

{numbered list}

#### ⚡ Checklist 规则（Review 时必查）

> **{一条可操作的规则，以「凡是…必须…」或「禁止…」开头}**
> {适用范围补充说明}
```

## 能力二：review —— Checklist 审查

**触发词**：「用错题本 check」、「checklist review」、「有没有踩过这个坑」、「check 历史 bug」

### 步骤

1. **读取** `ai-metadata/backend/common-mistakes.md`，提取所有 `⚡ Checklist 规则` 条目，
   编号和规则文本一起保留（用于输出引用）。

2. **读取目标代码**（用户指定的文件或本次变更的文件）。

3. **逐条匹配**：对每条规则，判断目标代码是否存在违反。

4. **输出结果**（见下方格式）。

### 输出格式

```
🗂️  Checklist Review — {目标文件/范围}
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

共 {N} 条规则

❌ 命中 {X} 条：
  [{编号}] {规则简述}
  → 位置：{文件}:{行号}
  → 问题：{具体说明}
  → 参考案例：{错题编号}

✅ 通过 {Y} 条（无问题）

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
{如有命中，附上修改建议}
```

如果所有规则都通过，输出一行：`✅ 全部 {N} 条规则通过，未发现历史 Bug 模式。`

---

## 注意事项

- 读写 `common-mistakes.md` 前，先确认文件存在，不存在则报错提示而不是创建空文件。
- Checklist 规则要能**机械地对照代码判断**，不要写「注意 X 问题」这种模糊表述。
- 错题本是积累性资产，只追加，不删除历史条目（用户明确要求除外）。
- add 时如果上下文信息不够，优先从当前对话的 Bug 分析中提取，再问用户补充。
