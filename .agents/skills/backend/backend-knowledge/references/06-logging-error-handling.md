# 06 - 公共能力：日志与错误处理

> 注意：日志与错误处理属于**跨层公共能力**，不是 pkg 开发子集。

## 日志打印指引

### 目标

- 让问题可定位
- 保留关键上下文
- 不泄露敏感信息

### 各层建议

- **Interface 层**：记录请求入口、参数摘要、请求 ID
- **App 层**：记录业务流程关键节点（开始/结束/分支）
- **Domain 层**：通常少打日志，重点是返回明确领域错误
- **Repo 层**：记录查询目标、耗时、失败原因（脱敏）

### 日志字段建议

- `request_id` / `trace_id`
- `user_id`（必要时脱敏）
- `module` / `action`
- `latency_ms`
- `err_code`

## 错误处理指引

### 分层策略

1. **Domain 错误**：表达业务约束失败（如状态不合法）
2. **Repo 错误**：表达存储失败（如记录不存在、唯一键冲突）
3. **App 错误**：组装上下文并决定是否转换错误类型
4. **Interface 错误**：映射为客户端可理解的响应

### 实践原则

- 返回可判定的错误类型或错误码
- 保留原始错误链路（wrap）
- 在边界层统一做错误到响应的映射
- 不把底层报错直接透传给用户

## 快速检查清单

- 是否记录了必要上下文？
- 是否避免打印敏感数据？
- 是否保留了原始错误链？
- 是否在 interface 层统一响应错误？
