根据 OpenSpec 变更实施任务。

**输入**：可选指定变更名（例如：`/opsx:apply add-auth`）。若未提供，可从对话上下文推断；若模糊或有歧义，**必须**提示用户从可用变更中选择。

**步骤**

1. **选择变更**

   若已提供名称则直接使用。否则：
   - 尝试从对话上下文推断用户提到的变更
   - 若仅有一个活跃变更，可自动选择
   - 若有歧义，执行 `openspec list --json` 获取可用变更，并使用 **AskUserQuestion 工具** 让用户选择

   始终明确提示："当前使用变更：<name>"，并告知可通过 `/opsx:apply <other>` 覆盖。

2. **检查状态并识别 schema**
   ```bash
   openspec status --change "<name>" --json
   ```
   解析 JSON 获取：
   - `schemaName`：当前使用的工作流（例如：`spec-driven`）
   - 哪个工件承载任务（通常是 `tasks`，但应以状态结果为准）

3. **获取 apply 指令**

   ```bash
   openspec instructions apply --change "<name>" --json
   ```

   返回内容包含：
   - `contextFiles`：工件 ID 到具体文件路径数组的映射（随 schema 而变）
   - 进度（总数、完成数、剩余数）
   - 任务列表及状态
   - 当前状态下的动态指导

   **处理状态：**
   - 若 `state: "blocked"`（缺失工件）：提示阻塞并建议使用 `/opsx:continue`
   - 若 `state: "all_done"`：提示已全部完成并建议归档
   - 否则继续实施

4. **读取上下文文件**

   读取 apply 指令返回的 `contextFiles` 中列出的每个路径。
   不同 schema 的文件可能不同：
   - **spec-driven**：通常为 proposal、specs、design、tasks
   - 其他 schema：严格按照 CLI 返回的 `contextFiles`

5. **展示当前进度**

   展示：
   - 当前 schema
   - 进度："N/M 任务已完成"
   - 剩余任务概览
   - CLI 返回的动态指导

6. **实施任务（循环直到完成或阻塞）**

   对每个待办任务：
   - 展示当前处理的任务
   - 完成所需代码改动
   - 改动保持最小且聚焦
   - 在任务文件中立即勾选完成：`- [ ]` → `- [x]`
   - 继续处理下一项

   **以下情况暂停：**
   - 任务语义不清：先询问澄清
   - 实施中暴露设计问题：建议更新工件
   - 出现错误或阻塞：报告并等待指示
   - 用户中断

7. **完成或暂停后展示状态**

   展示：
   - 本次会话完成的任务
   - 总体进度："N/M 任务已完成"
   - 若全部完成：建议归档
   - 若暂停：说明原因并等待下一步

8. **更新知识图谱（最终步骤）**

   无论全部完成还是暂停，均在最后执行 `/graphify update`，将本轮变更同步到项目知识图谱。
   此步骤为所有 apply 流程的**固定收尾动作**，不可跳过。

**实施过程输出示例**

```text
## 正在实施：<change-name>（schema: <schema-name>）

处理任务 3/7：<任务描述>
[...实施中...]
✓ 任务完成

处理任务 4/7：<任务描述>
[...实施中...]
✓ 任务完成
```

**完成输出示例**

```text
## 实施完成

**变更：** <change-name>
**Schema：** <schema-name>
**进度：** 7/7 任务完成 ✓

### 本次完成
- [x] 任务 1
- [x] 任务 2
...

全部任务已完成。可执行 `/opsx:archive` 进行归档。
```

**暂停输出示例（遇到问题）**

```text
## 实施已暂停

**变更：** <change-name>
**Schema：** <schema-name>
**进度：** 4/7 任务完成

### 遇到的问题
<问题描述>

**可选操作：**
1. <选项 1>
2. <选项 2>
3. 其他方案

你希望如何继续？
```

**护栏**
- 持续推进任务，直到全部完成或出现明确阻塞
- 开始前必须先读取所有上下文文件（来自 apply 指令输出）
- 若任务有歧义，先询问后实现
- 若实现暴露问题，先暂停并建议更新工件
- 代码修改保持最小、范围清晰
- 每完成一项，立即更新任务勾选状态
- 遇到错误、阻塞或需求不清时暂停，不要猜测
- 使用 CLI 输出的 `contextFiles`，不要假设固定文件名

**流式工作集成**

该技能支持“围绕变更执行动作”的流式模式：

- **可随时调用**：工件未全完、部分已实现或与其他动作交错时都可调用
- **允许工件回写**：若实现中出现设计问题，应建议更新工件，不受严格阶段限制
