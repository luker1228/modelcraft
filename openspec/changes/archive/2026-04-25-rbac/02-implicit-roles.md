# 内置隐式角色

> 对应总纲：`00-rbac-overview.md`
> 覆盖内容：隐式角色定义、落库规范、运行时注入规则、承载能力范围

---

## 问题背景

系统中存在一类"所有已登录用户天然拥有的基础能力"，例如：

- 查看自己的账号信息
- 修改自己的个人资料
- 修改自己的密码
- 查看自己的通知
- 访问登录后的基础页面

这些能力属于**基础自服务能力**，不属于业务岗位权限，不应由管理员手工分配，但又需要纳入权限体系统一管理和审计。

---

## 解法：内置隐式角色

系统采用 **内置隐式角色（Implicit Role）** 模型：

```
SYSTEM_AUTHENTICATED_USER（示例）
  ├── 定义落库（Role.isImplicit = true）
  ├── 在系统中可见、可审计
  ├── 不作为普通角色手工分配
  ├── 用户与该角色的关系不逐条落库
  └── 鉴权时由系统自动注入
```

### 四条规则（总结）

> **角色落库，关系隐式；定义可见，授予自动。**

| 规则 | 含义 |
|------|------|
| **角色落库** | 内置隐式角色写入 `roles` 表，`isImplicit = true` |
| **关系隐式** | `user_roles` 表中**不**为每个用户插入隐式角色关联行 |
| **定义可见** | 权限管理界面可以查看和修改隐式角色的权限包配置 |
| **授予自动** | 鉴权时，系统自动将隐式角色注入当前用户的角色集合 |

---

## 与普通角色的对比

| 维度 | 普通角色 | 内置隐式角色 |
|------|----------|-------------|
| 落库 | 是 | 是 |
| 用户关系落库 | 是（`user_roles` 表） | 否 |
| 手工分配 | 是 | 否 |
| 鉴权注入方式 | 查 `user_roles` | 系统自动注入 |
| 审计可见性 | 可见 | 可见 |
| 修改权限包 | 可以 | 可以（受限，不可删除内置角色） |

---

## 承载能力范围

内置隐式角色**只承载基础自服务能力**：

✅ 适合放入隐式角色：
- 查看/修改自己的账号信息
- 修改自己的密码
- 查看自己的通知
- 访问登录后的基础导航页

❌ 不适合放入隐式角色：
- 业务岗位权限（如"订单管理""客户查看"）
- 需要审批或限制分配的能力
- 应由管理员按角色分配的能力

---

## 鉴权时的注入规则

见 [03-auth-flow.md](./03-auth-flow.md)，鉴权第三步：

```
Step 3: 自动加入隐式角色绑定的权限包
  → 查询 Role WHERE isImplicit = true
  → 取其关联的所有权限包，合并进有效权限集
```

此步骤对所有已认证用户执行，无需任何 `user_roles` 查询。
