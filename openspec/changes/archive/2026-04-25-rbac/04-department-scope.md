# 部门与数据范围

> 对应总纲：`00-rbac-overview.md`
> 覆盖内容：部门在权限体系中的定位、行策略展开规则、与授权对象的边界

---

## 部门的职责定位

**部门不是正式授权载体。**

部门在本方案中的核心职责是：**作为行权限（数据范围）计算的上下文来源。**

```
授权对象（用户 / 角色）决定：拥有什么权限
部门                  决定：权限作用到哪部分数据
```

### 职责边界

| 职责 | 部门 | 用户/角色 |
|------|------|----------|
| 主授权（"给谁授什么权限"） | ❌ | ✅ |
| 日常权限分配入口 | ❌ | ✅ |
| 提供行权限数据范围 | ✅ | ❌ |
| 支撑 DEPT / DEPT_AND_CHILDREN 展开 | ✅ | ❌ |

---

## 行策略与部门的关系

权限点的行策略（`rowScope`）通过以下方式与部门交互：

### `SELF`（不依赖部门）

```
行策略: SELF
计算规则: WHERE created_by = current_user_id
依赖: 当前用户 ID，无需部门信息
```

### `DEPT`（依赖当前部门）

```
行策略: DEPT
计算规则: WHERE dept_id = current_user_dept_id
依赖: 用户当前所在部门 ID
展开: 单个部门，不含下级
```

### `DEPT_AND_CHILDREN`（依赖部门树）

```
行策略: DEPT_AND_CHILDREN
计算规则: WHERE dept_id IN (current_dept_id, child_dept_ids...)
依赖: 用户当前部门 + 递归获取所有下级部门 ID
展开: 当前部门 + 全部下级部门（递归）
```

### `ALL`（不依赖部门）

```
行策略: ALL
计算规则: 不加 WHERE 条件（全量）
依赖: 无
```

## 多部门归属（扩展说明）

第一版假设用户属于**单一部门**。

若后续需要支持用户归属多个部门，行策略展开规则调整为：

```
DEPT:               WHERE dept_id IN (dept_id_1, dept_id_2, ...)
DEPT_AND_CHILDREN:  WHERE dept_id IN (所有归属部门及其子部门集合)
```

当前版本不实现多部门，此处仅作预留说明。

---

## 部门 vs. 授权载体（总结）

| 场景 | 使用什么 |
|------|----------|
| "给销售团队授予订单查看权限" | 创建"销售"角色 → 关联权限包，不使用部门授权 |
| "只允许看本部门的订单" | 权限点 rowScope = DEPT，由部门提供范围过滤 |
| "下级领导可以审批下属工单" | 权限点 rowScope = DEPT_AND_CHILDREN，部门树展开 |

**结论：部门是数据范围的上下文，不是授权的主体。所有权限分配仍通过用户和角色完成。**
