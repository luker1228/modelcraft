# 用户认证（登录 & 注册）

## 是什么问题

现有认证方案基于 AuthProvider，但实际只用到登录和注册两个功能，AuthProvider 的部署和维护成本远超实际需求。目标是用自研轻量方案替换 AuthProvider，降低系统复杂度。

## 目标用户

国内开发者，使用手机号作为主要身份标识。

## 用户故事

- 作为新用户，我希望用手机号 + 用户名 + 密码注册账号，以便开始使用 ModelCraft。
- 作为已注册用户，我希望用手机号 + 密码登录，以便访问我的工作空间。
- 作为已注册用户，我希望用用户名 + 密码登录，以便访问我的工作空间。

## 核心功能

**注册**
- 输入手机号 + 用户名 + 密码（+ 确认密码），提交后创建账号
- 手机号格式校验（11 位国内号码），唯一性校验
- 用户名规则：英文字母/数字/`_`/`-`，不能数字开头，长度 3–32 位，全局唯一，注册后不可修改
- 用户名过滤保留字（admin、root、system 等），维护静态黑名单
- 密码强度要求：最少 8 位
- 注册成功后：`displayName` 默认等于 `userName`；自动创建初始 org（orgName 随机生成）；直接跳转工作区

**登录**
- 支持手机号 + 密码、用户名 + 密码两种方式
- 验证通过后签发 JWT Token（有效期 7 天）
- 登录成功后跳转到 workspace 页
- 错误提示：账号不存在 / 密码错误

## 密码输入交互

- 密码输入框提供显示/隐藏切换图标（眼睛），点击切换明文/密文
- 注册页的「密码」和「确认密码」均需支持

## 不做什么

- 不做短信验证码登录
- 不做第三方登录（微信、GitHub 等）
- 不做忘记密码 / 重置密码
- 不做邮箱注册
- 不做图形验证码 / 人机校验
- 不做多端 Token 管理 / 强制下线
- 不迁移 AuthProvider 的其他能力（OAuth Server、权限管理等均不需要）
- 无存量用户数据，不需要数据迁移

## 替换 AuthProvider 的影响范围

- 后端：移除 AuthProvider SDK 依赖，自行实现 JWT 签发与校验
- 前端：替换登录/注册页面，调整 Token 存储与请求头携带方式
- 基础设施：AuthProvider 服务可下线，减少一个部署依赖
