## Context

当前 EndUser 后端模型以 `org + project` 为账号归属，导致同一组织内跨项目复用账号需要重复建号。`enduser-v2` 要求将账号归属提升为 Org 级，并通过单独的“项目访问授权关系”表达可访问项目与权限包。运行时仍需按项目签发 JWT，以保持现有 RLS（依赖 `projectSlug`）行为不变。

## Goals / Non-Goals

**Goals:**
- 将 EndUser 主账号模型切换为 Org 级唯一（`UNIQUE(org_name, username)`）。
- 引入 EndUser-Project 访问关系模型，支持授权、撤销、查询、权限包绑定。
- 将登录改为一次签发：认证通过后直接返回 JWT，并返回可访问项目全集供用户自由选择。
- 保持 JWT 与 RLS 兼容：JWT 继续携带 `projectSlug`。
- 完成数据库与仓储层迁移，确保数据可平滑演进。

**Non-Goals:**
- 不改造 RLS 规则与 SQL 执行策略。
- 不引入新的认证协议（如 OAuth/OIDC）。
- 不在本变更中重构前端交互细节（仅定义后端契约）。

## Decisions

1. **账号归属改为 Org 级，访问权限独立建模**  
   - 方案：`EndUser` 去除项目归属字段，仅保留 Org 维度；新增 `EndUserProjectAccess` 记录项目访问。  
   - 备选：继续沿用 `org+project+username` 并增加“主账号映射表”。  
   - 取舍：独立访问关系更符合“一号多项目”语义，查询与审计边界更清晰。

2. **GraphQL 按上下文分层：Org 管账号，Project 管授权**  
   - 方案：Org schema 提供账号生命周期操作；Project schema 提供访问授权操作。  
   - 备选：全部放入 Org schema。  
   - 取舍：授权操作天然依赖项目上下文，放在 Project schema 可减少跨域参数传递与误用。

3. **登录成功一次签发令牌，并返回全部可访问项目**  
   - 方案：`login` 在凭据校验通过后直接签发 JWT，同时返回该 EndUser 在组织下有权限的全部项目列表；登录后用户可在已授权项目中自由选择当前项目。  
   - 备选：拆分为 `login + select-project` 两次交互后再签发令牌。  
   - 取舍：满足“一次签发即可使用”的产品约束，同时保留项目级权限边界。

4. **迁移策略采用“结构迁移 + 数据回填 + 双向校验”**  
   - 方案：先加新表与新约束，再回填访问关系，最后移除旧字段。  
   - 备选：一次性切表。  
   - 取舍：分阶段可控，便于回滚与数据一致性校验。

## Risks / Trade-offs

- **[风险] 历史数据存在同组织同用户名跨项目重复记录** → **缓解**：按 `created_at` 选主记录，其他记录合并为访问关系并输出迁移审计报告。  
- **[风险] 登录响应承载项目列表，调用方解析逻辑需调整** → **缓解**：保持错误码稳定，补充响应契约测试与兼容字段校验。  
- **[风险] 删除旧 `project_slug` 字段后兼容调用报错** → **缓解**：在发布窗口前完成全链路契约扫描并配套集成测试。  
- **[权衡] 模型更清晰但查询路径变长** → **缓解**：为访问关系加联合索引（`org_name, project_slug, end_user_id`）。

## Migration Plan

1. 数据库增量迁移：创建 `end_user_project_access`，新增必要索引与外键。  
2. 数据回填：从历史账号/角色关联中生成访问关系。  
3. 服务端切换：仓储与应用层改读写新模型；GraphQL/BFF 切到新契约。  
4. 灰度校验：比对旧流程与新流程的登录成功率、授权结果一致性。  
5. 清理收口：删除旧项目绑定字段与废弃查询路径。

- 登录后项目切换是否通过独立上下文接口实现，还是通过现有会话更新机制承载？
