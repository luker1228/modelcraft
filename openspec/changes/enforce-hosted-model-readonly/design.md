## Context

系统需要同时支持自建模型与导入模型，但当前缺少统一的“模型来源能力边界”。结果是托管模型可能被误执行结构变更或数据写入，违反“外部托管源仅受管不可改”的约束。

本次变更要求把模型类型作为强约束输入到后端判定链路：
- 创建模型 → 必须是 `SELF_BUILT`
- 导入模型 → 必须是 `MANAGED`
- `MANAGED` 模型 → 结构冻结 + 数据只读

## Goals / Non-Goals

**Goals:**
- 在模型创建/导入入口落定模型类型，保证来源语义不可歧义。
- 在后端统一拦截托管模型的 DDL、字段增删改、数据写操作。
- 保留托管模型的读取能力（结构查询、字段查询、数据查询）。
- 提供一致错误码/错误信息，便于前端禁用按钮与提示原因。

**Non-Goals:**
- 不改变自建模型现有可编辑行为。
- 不引入新的模型类型。
- 不在本变更中扩展细粒度字段级权限体系。
- 不处理托管源外部发生的结构漂移同步机制。

## Decisions

1. **以模型类型作为单一判定条件**
   - 决策：所有写路径在应用层统一读取模型类型并执行能力判定。
   - 原因：避免在多个 resolver/handler 重复分散判断，降低漏拦截风险。
   - 备选：仅在接口层校验。放弃原因：内部调用链可能绕过接口层。

2. **创建/导入路径强绑定类型**
   - 决策：创建模型接口不接受可选类型，后端固定写入 `SELF_BUILT`；导入流程固定写入 `MANAGED`。
   - 原因：防止调用方误传或恶意传入，保证来源可信。
   - 备选：允许前端传类型并校验。放弃原因：增加协议复杂度且无收益。

3. **托管模型执行“默认拒绝写”策略**
   - 决策：对 `MANAGED` 模型统一拒绝：DDL、字段增删改、数据增删改。
   - 原因：规则简单、可审计、易测试，且与业务目标一致。
   - 备选：白名单允许部分写操作。放弃原因：边界复杂，易出现策略漏洞。

4. **错误语义标准化**
   - 决策：返回稳定错误类型（如 `MANAGED_MODEL_READONLY`）与可读 message。
   - 原因：前端可据此进行禁用态与统一提示，测试可稳定断言。
   - 备选：复用通用权限错误。放弃原因：无法区分“无权限”与“模型类型限制”。

## Risks / Trade-offs

- **[风险] 漏拦截内部写入口** → **缓解**：在应用服务层集中守卫，并增加覆盖关键写路径的测试。
- **[风险] 历史接口错误文案不一致** → **缓解**：统一错误构造器与错误码映射。
- **[权衡] 规则偏严格，短期降低灵活性** → **缓解**：后续若有业务需求，再通过显式能力白名单单独评审。

## Migration Plan

1. 新增模型类型判定与只读守卫。
2. 接入创建/导入流程，固化类型赋值。
3. 为字段、DDL、数据写入口增加托管模型阻断。
4. 补充测试用例（允许读取、拒绝写入）。
5. 分阶段发布：先后端守卫，再前端禁用入口。
6. 回滚策略：可按模块回滚守卫逻辑，不影响现有数据结构。
