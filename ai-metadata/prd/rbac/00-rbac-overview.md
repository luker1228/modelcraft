# 权限模型总览（RBAC）

> 本目录覆盖 ModelCraft **开发者侧**管理权限的完整设计：谁能对哪张 Model 做什么操作、能看哪些行和列。
> EndUser 侧的数据隔离由 `rls/` 目录负责。

## 定位关系

```
end-user-auth/          ← 认证层：谁能登录，会话如何隔离
    │
    ▼
rbac/                   ← 鉴权层：登录后能做什么（本目录）
    ├── 权限点（最小能力单元，描述表×动作×列策略×行策略）
    ├── 权限包（多个权限点的组合，正式授权单位）
    └── 授权对象（用户 / 角色）
    │
    ▼
rls/                    ← 数据隔离层：Runtime 查询自动注入 WHERE
                           依赖 EndUserRef 字段 + ModelRLSPolicy
```

| 层 | 目录 | 解决的问题 |
|----|------|-----------|
| 认证 | `end-user-auth/` | 终端用户如何登录，Cookie 隔离，会话管理 |
| 鉴权 | `rbac/` | 已登录用户能做什么，资源×动作×列策略×行策略授权 |
| 数据隔离 | `rls/` | Runtime 查询自动按 owner 字段过滤数据行 |

---

## 核心设计原则（最终方案）

系统第一版采用以下三条**不可妥协**的核心原则：

1. **只做 allow，不做 deny** — 默认拒绝，显式允许；不引入 deny 规则，不处理冲突优先级
2. **只授权限包，不直授权限点** — 权限点负责定义，权限包负责授权；用户/角色只能关联权限包
3. **只以用户和角色为授权载体** — 部门不承担主授权职责，只参与行策略（RLS）的数据范围计算

一句话版：

> **只做 allow；只给用户和角色授权；只授权限包；部门只管行策略数据范围；隐式角色落库定义、运行时自动生效。**

## 不做什么（Out of scope，第一版）

| 排除项 | 原因 |
|--------|------|
| **deny 机制** | 避免多来源冲突优先级复杂性 |
| **直接授权限点** | 避免零散补丁式权限管理 |
| **部门作为主授权对象** | 部门只参与数据范围计算 |
| **权限审批流** | 第一版只做配置，不做工作流 |
| **细粒度字段级加密** | 超出本期范围 |
